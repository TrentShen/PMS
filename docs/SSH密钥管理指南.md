# SSH 密钥管理指南（给 Kimi Code CLI 使用）

> 本指南面向非技术人员。按步骤复制粘贴即可，不需要理解技术原理。

---

## 一、基本原则

1. **一个项目 = 一对 SSH 密钥**。不要把同一个私钥用在多个项目上。
2. **公钥可以发给运维，私钥绝对不能外传**。
3. 私钥只保存在你自己的电脑上（`~/.ssh/` 目录）。
4. 如果私钥丢失或泄露，需要重新生成并通知运维更换服务器上的公钥。

---

## 二、快速操作

### 步骤 1：让 Kimi 生成项目专用密钥

把下面这段话复制给对应项目的 Kimi：

```text
请帮我为 [项目名称] 生成一对 SSH 部署密钥，并把公钥发给我。私钥请保存在 ~/.ssh/id_[项目简称]。
```

例如：
- PMS → `id_pms`
- 企业微信机器人 → `id_wecom_bot`
- 招聘系统 → `id_recruit`

### 步骤 2：把 Kimi 给你的公钥发给运维

复制 Kimi 返回的公钥，并附上以下说明：

```text
运维同学好，
请把下面这行公钥添加到 [服务器IP] 的部署用户 ~/.ssh/authorized_keys 文件中。

[粘贴公钥]

配置完成后请告诉我：
1. 服务器 IP：
2. SSH 端口（默认 22）：
3. 登录用户名：
4. 项目部署目录：
5. 该用户是否有 docker / systemctl / alembic 等操作权限：
```

### 步骤 3：把运维回复的信息告诉 Kimi

Kimi 拿到 IP、用户名、端口、部署目录后，会先测试 SSH 连通，再开始部署。

---

## 三、为什么要这样做？

| 概念 | 通俗解释 |
|------|---------|
| 公钥 | 像一把“门锁”，可以公开给服务器 |
| 私钥 | 像一把“钥匙”，只能由你自己保管 |
| 一个项目一把钥匙 | 避免一把钥匙丢了，所有项目都受影响 |

---

## 四、安全注意事项

1. **不要把私钥文件通过微信、邮件、网盘发送给任何人**。
2. **不要把私钥复制到聊天窗口**。
3. 如果电脑丢失、重装系统、或怀疑私钥泄露，立即让运维删除服务器上对应的公钥。
4. 私钥文件默认权限是 `600`，不要手动修改权限。

---

## 五、常见问题

**Q：私钥存在哪里？**
A：存在你当前电脑的 `~/.ssh/` 目录下，例如 `/Users/你的名字/.ssh/id_pms`。

**Q：换电脑了怎么办？**
A：需要重新生成密钥对，并重新让运维配置公钥。

**Q：多个 Kimi 项目能共用同一把私钥吗？**
A：技术上可以，但不推荐。建议每个项目独立密钥，方便权限管理和问题追溯。

**Q：公钥配错了怎么办？**
A：让运维从 `~/.ssh/authorized_keys` 里删除错误的那一行，重新配正确的公钥。

## 六、运维侧参考命令

如果运维需要一次性为多个项目配置公钥，可以在服务器上执行：

```bash
mkdir -p ~/.ssh && chmod 700 ~/.ssh

# 项目 A
echo "ssh-ed25519 AAAA... 项目A" >> ~/.ssh/authorized_keys

# 项目 B
echo "ssh-ed25519 AAAA... 项目B" >> ~/.ssh/authorized_keys

# 项目 C
echo "ssh-ed25519 AAAA... 项目C" >> ~/.ssh/authorized_keys

chmod 600 ~/.ssh/authorized_keys
```
