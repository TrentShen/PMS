## 一、系统整体评价

### 1.1 亮点

- **骨架完整度高**：目标→自评→互评→上级评估→校准→审批→反馈→发布全链路代码到位，试用期管理已落地
- **工程规范有章法**：HARNESS v2.1 的 P-C-R-V 循环 + 6 道质量门禁 + RULE-012 行为基线 + 错误登记册
- **后端测试有保障**：50 个测试全绿，覆盖评分校验、权限、happy path
- **企微集成完整**：OAuth 免登、通讯录同步、应用消息推送、移动端适配
- **文档体系完善**：PRD、技术方案、使用手册、差距评估、部署指南一应俱全

### 1.2 核心差距

| 层面 | 现状 | 差距 |
|------|------|------|
| 功能完备性 | 绩效周期 + 试用期已覆盖主路径 | 缺历史绩效结果导入、人才盘点、绩效趋势分析 |

## 二、改进建议（对标市面绩效系统）

### 2.1 历史绩效结果导入（P0 — 当前缺失）

**现状**：系统只有绩效目标（Objective）的 Excel 导入，没有历史绩效结果（评分、等级、价值观）的导入功能。

**建议方案**：

| 功能点 | 实现方案 | 优先级 |
|--------|----------|--------|
| 历史绩效结果导入模板 | 字段：员工ID、姓名、周期名称、业绩分、业绩等级、价值观等级（三维度）、上级评语摘要 | P0 |
| 导入时自动创建历史 Cycle | 以 `status=published` 的只读 Cycle 存储，不参与当前流程 | P0 |
| 导入校验规则 | 员工ID存在性、业绩分0.25分段、价值观枚举值、周期唯一性 | P0 |
| 上级评估页历史展示 | 确保导入的历史数据在评估页 `history_perf` 展示 | P1 |
| 历史绩效趋势图 | 个人视角折线图；管理视角部门分布趋势 | P2 |

**关键设计**：导入的历史 Cycle 应为"只读快照"——不关联 Objective 表，只存评估结果，最轻量且不污染当前流程。

### 2.2 绩效周期管理补全

#### 2.2.1 流程开关未生效（阻塞级 — B-4）

`enable_*` 字段存了但前端不读、后端不拦截。HR 配置"简版考核"后员工仍看到完整流程。

**修复方案**：
- 后端：在 `evaluations.py`、`peer.py`、`calibration.py`、`feedback.py` 写接口入口加 `cycle.enable_*` 检查
- 前端：根据 `cycle.enable_*` 控制导航菜单和按钮显示/隐藏

#### 2.2.2 关键节点通知缺失（高优先级 — H-1）

当前只有催办和截止提醒，缺以下通知：
- 周期启动 → 全员
- Leader 确认互评名单 → 互评人
- 校准提交 → HR/CEO
- 审批驳回 → Leader
- 结果发布 → 员工

#### 2.2.3 发布前未校验反馈完成（阻塞级 — B-5）

`publish_cycle` 只检查最终分数，不检查面谈确认。需在 `enable_feedback=True` 时校验所有参与人 `FeedbackRecord.confirm_status != pending`。

#### 2.2.4 匿名评价可见性违背 PRD（阻塞级 — B-3）

`peer.py:524` 把 `direct_leader` 纳入匿名评价可见范围，需移除，仅保留 `hrbp/super_admin/dept_leader`。

#### 2.2.5 其他高优先级问题

| 编号 | 问题 | 修复方向 |
|------|------|----------|
| B-1 | 生产 HTTPS/端口配置与仓库不匹配 | 补充 443/SSL 配置 |
| B-2 | 企微 agentid 为空时消息发送崩溃 | 发送前校验配置 |
| H-3 | 匿名评价仍为单维度 | 新增三维度字段 |
| H-4 | 校准后价值观未继承 | 未传入维度自动继承上级评估 |
| H-5 | feedback.py N+1 查询 | 批量查询再映射 |
| H-6 | 前端 any 类型 55 处 | 逐步替换为具体类型 |
| H-7 | 移动端复杂页面体验差 | 步骤引导、卡片化、放大触摸区域 |

## 三、Harness 工程改进建议

### 3.1 当前亮点

1. **RULE-012 四条铁律**：直击 AI 编码核心痛点
2. **错误登记册**：沉淀"踩过的坑"
3. **P-C-R-V 循环**：有明确验证环节
4. **前后端契约对照表**：命中最高频 Bug 来源

### 3.2 改进建议

#### 3.2.1 增加自动化契约校验脚本（最大短板）

当前 Verify 靠"AI 运行测试 + 人工确认"，无自动化契约校验。errors.md 中 4 个错误有 3 个是前后端字段不一致。

**建议**：新增 `scripts/check_contract.py`：
1. 扫描后端所有 Pydantic Schema 字段名和类型
2. 扫描前端所有 API 调用的 payload 字段名和类型
3. 对比差异，输出 mismatch 报告
4. 作为 Gate 1 的自动执行项

#### 3.2.2 错误登记册增加"规则触发机制"

新任务开始时，AI 应主动检查 errors.md 中是否有相关历史错误，如有则在 Checklist 中增加对应校验项。

#### 3.2.3 规则文件覆盖面扩展

| 建议新规则 | 防范的问题 |
|-----------|-----------|
| RULE-006 流程开关一致性 | enable_* 字段必须前后端同步读取 |
| RULE-007 通知触发完整性 | 关键状态变更必须检查是否需要发企微消息 |
| RULE-008 N+1 查询检查 | list 接口内禁止循环查询数据库 |
| RULE-009 环境隔离 | 生产环境禁止暴露调试接口 |
| RULE-010 权限可见性审计 | 新增可见性判断时必须对照 PRD 权限矩阵 |

#### 3.2.4 增加结构化 Review 自检清单

```markdown
## Review 自检清单（每次 Code 完成后必须逐项打勾）

- [ ] 前后端字段对照表已核对（RULE-001）
- [ ] 新增接口有权限校验（RULE-002）
- [ ] 涉及多表操作的事务边界正确（RULE-003）
- [ ] 无 datetime.utcnow()（RULE-004）
- [ ] 新增模型已在 __init__.py 导入（RULE-005）
- [ ] 流程开关是否需要同步（如涉及）
- [ ] 关键节点是否需要发通知（如涉及状态变更）
- [ ] 无 N+1 查询（list 接口内无循环查询）
- [ ] errors.md 中相关历史错误已检查
- [ ] 无新的 console.log / TODO / FIXME
```

#### 3.2.5 HARNESS 架构演进建议

```
当前：Plan → Code → Review → Verify → (发现问题回到 Plan/Code)

建议：
Plan → Code → Review(自检清单) → Verify(测试+契约校验脚本) → Deploy(灰度) → Monitor(错误日志)
  ↑                                                                    |
  └────────── 定期回顾（errors.md + 规则更新 + 技术债清理）──────────────┘
```

增加 Deploy（灰度）和 Monitor（监控）环节：
- Deploy：Docker Compose 滚动更新，先灰度 1-2 人验证
- Monitor：loguru 日志 + 错误告警（如企微消息失败率 > 10% 告警）

## 四、Kimi 协作 Loop 改进建议

### 4.1 当前局限

1. **人在闭环里**：每轮迭代需人发现问题、描述问题、下指令
2. **上下文丢失**：新会话从零开始，之前的设计决策不一定携带
3. **无自动化反馈**：代码改完无自动回归检查

### 4.2 改进方案

#### 4.2.1 每周迭代回顾机制

在 `docs/迭代回顾-YYYYMMDD.md` 中用结构化模板记录：
- 本周变更清单
- 自动化检查结果（测试/类型/构建/契约/安全）
- 已知问题跟踪
- 技术债清单
- 下周计划

#### 4.2.2 对抗性 Review 环节

定期让 Kimi 切换为"挑剔的 Senior Engineer"角色做 Code Review：
- 前后端字段一致性
- 权限漏洞
- 性能问题
- 安全隐患
- PRD 合规性

#### 4.2.3 技术债看板

在 `docs/技术债看板.md` 维护所有非阻塞性改进项，按影响排序处理。

#### 4.2.4 跨会话上下文传递

利用 `.workbuddy/memory/` 记录每次重要迭代的完成情况、遗留问题、下一步优先级。

#### 4.2.5 CI/CD 轻量化闭环

增加 GitHub Actions / GitLab CI：
- 后端 pytest
- 前端 tsc --noEmit + npm run build
- 契约校验脚本

让 Kimi 引用 CI 结果作为 Verify 的客观证据。
