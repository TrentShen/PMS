# 技术方案：目标周期与评估周期解耦（B+ 方案）

> 状态：已确认，进入实施  
> 目标版本：PMS V1.0  
> 作者：Kimi  
> 日期：2026-06-29

---

## 一、背景与问题

当前 `PerformanceCycle` 把「目标制定」「目标执行」「绩效评估」绑定在同一条流程线上：

```
draft → in_progress（目标制定 + 自评 + 互评 + 上级评估 + 校准） → published → closed
```

业务痛点：

1. 无法同时开展「评估当前周期」和「制定下一周期目标」。
2. 每年两次的绩效窗口期（每次约 1 个月）需要员工同时处理「上半年评估」和「下半年目标制定」，系统不支持并行。
3. 目标一旦进入评估流程就被锁死，没有独立的目标管理生命周期。

---

## 二、设计目标

1. 支持目标周期与绩效评估周期独立存在、并行运行。
2. 绩效评估可以引用对应目标周期的目标作为参考，但不以目标完成率直接决定绩效结果。
3. 半年度节奏不变：目标周期半年一次，评估周期半年一次，两者有 1 个月左右重叠期。
4. 系统不跟踪目标实时执行进度，只关注「目标制定」和「绩效评估」两个任务。
5. 目标周期何时开始执行，由 HR 根据审批进度在系统中确认。
6. 改造可控，2-3 周内可落地；架构上保留未来向完全解耦（方案 C）演进的空间。

---

## 三、业务节奏（已确认）

```
2024.12 - 2025.01   制定 2025 上半年目标（ObjectiveCycle: 2025H1，draft）
2025.01             2025 上半年目标启动（active）
2025.01 - 2025.06   执行 2025 上半年目标（系统不跟踪进度）
2025.06 - 2025.07   评估 2025 上半年目标（PerformanceCycle: 2025H1 eval，in_progress）
                    + 制定 2025 下半年目标（ObjectiveCycle: 2025H2，draft）
2025.07             2025 上半年目标完成（completed）
                    2025 下半年目标启动（active）
2025.07 - 2025.12   执行 2025 下半年目标
2025.12 - 2026.01   评估 2025 下半年目标 + 制定 2026 上半年目标
```

重叠期员工首页需同时展示：
- 当前周期评估任务（如：2025 上半年绩效评估）
- 下个周期目标制定任务（如：2025 下半年目标制定）

## 四、核心设计

### 4.1 周期关系

```
ObjectiveCycle（目标周期）
├─ 2025 上半年目标
│   ├─ draft（2024.12-2025.01 制定）
│   ├─ active（2025.01-2025.06 执行）
│   └─ completed（2025.07 结束）
│
└─ 2025 下半年目标
    ├─ draft（2025.06-2025.07 制定，与上半年评估重叠）
    ├─ active（2025.07-2025.12 执行）
    └─ completed（2025.12 结束）

PerformanceCycle（绩效评估周期）
└─ 2025 上半年绩效评估
    ├─ objective_cycle_id → 2025 上半年目标
    └─ in_progress（2025.06-2025.07）
```

### 4.2 关键原则

| 原则 | 说明 |
|------|------|
| 目标周期独立 | `ObjectiveCycle` 有自己的状态机：draft → active → completed |
| 评估周期引用目标 | `PerformanceCycle.objective_cycle_id` 指向被评估的目标周期 |
| 1 对 1 关联 | 一个评估周期评估一个目标周期；一个目标周期可被后续评估周期引用 |
| 重叠期支持 | 系统允许一个 active 目标周期、一个 draft 目标周期、一个 in_progress 评估周期同时存在 |
| 目标仅作参考 | 评估表单中展示目标，但不参与绩效得分计算 |
| 不跟踪执行进度 | 系统不关心目标完成百分比，只关心「是否已制定/审批/生效」 |
| HR 控制生效时间 | 目标周期从 draft → active 由 HR 手动确认 |

## 五、数据模型变更

### 5.1 新增 `objective_cycle` 表

```python
class ObjectiveCycle(SQLModel, table=True):
    __tablename__ = "objective_cycle"

    id: int | None = Field(default=None, primary_key=True)
    name: str = Field(max_length=128)                      # 如：2025 上半年目标
    start_date: date                                        # 目标执行开始日期
    end_date: date                                          # 目标执行结束日期
    # 状态机：draft -> active -> completed
    status: str = Field(default="draft", max_length=32, index=True)
    created_by: str = Field(max_length=64)                  # wecom_userid
    created_at: datetime = Field(default_factory=lambda: datetime.now(timezone.utc))
    completed_at: datetime | None = None
```

### 5.2 修改 `objective` 表

现有：

```python
cycle_id: int = Field(foreign_key="performance_cycle.id", index=True)
```

改为：

```python
objective_cycle_id: int = Field(foreign_key="objective_cycle.id", index=True)
# 评估时冗余关联，便于查询（可选，评估启动时回填）
performance_cycle_id: int | None = Field(
    default=None, foreign_key="performance_cycle.id", index=True, nullable=True
)
```

### 5.3 修改 `performance_cycle` 表

新增字段：

```python
objective_cycle_id: int | None = Field(
    default=None, foreign_key="objective_cycle.id", index=True, nullable=True
)
```

### 5.4 修改 `objective_revision` 表

现有 `cycle_id` 指向 `performance_cycle.id`，改为指向 `objective_cycle.id`：

```python
objective_cycle_id: int = Field(foreign_key="objective_cycle.id", index=True)
```

### 5.5 数据迁移策略

1. 创建 `objective_cycle` 表。
2. 为每个现有 `performance_cycle` 生成一个对应的 `objective_cycle`：
   - name = `${performance_cycle.name}`（或去掉"考核"替换为"目标"）
   - start_date / end_date 与 performance_cycle 相同
   - status 根据 performance_cycle.status 映射：
     - draft → draft
     - in_progress → active
     - published / closed → completed
3. 把现有 `objective` 的 `cycle_id` 迁移到 `objective_cycle_id`。
4. 把现有 `objective_revision` 的 `cycle_id` 迁移到 `objective_cycle_id`。
5. 在 `performance_cycle` 中回填对应的 `objective_cycle_id`。

> 由于生产数据目前不可达，迁移脚本需要在本地和测试环境验证后，再在恢复后的生产环境执行。

## 八、状态机

### 8.1 ObjectiveCycle 状态机

```
draft（制定中）──start──► active（执行中）──complete──► completed（已结束）
  │
  └──delete──► （删除）
```

### 8.2 PerformanceCycle 状态机（不变）

```
draft ──start──► in_progress ──publish──► published ──close──► closed
```

### 8.3 重叠期允许的状态组合

| ObjectiveCycle A | ObjectiveCycle B | PerformanceCycle | 业务含义 |
|------------------|------------------|------------------|---------|
| 2025H1 active | 2025H2 draft | 2025H1 eval in_progress | 评上半年 + 定下半年 |
| 2025H2 active | 2026H1 draft | 2025H2 eval in_progress | 评下半年 + 定上半年 |
| 2025H2 draft | — | — | 只制定目标 |
| 2025H2 active | — | — | 只执行目标 |

## 十一、风险与应对

| 风险 | 影响 | 应对 |
|------|------|------|
| 现有测试用例大量依赖 `cycle_id` 关联目标 | 高 | 同步更新测试 fixtures 和断言 |
| 生产数据迁移复杂 | 中 | 编写独立迁移脚本，先在本地/测试环境验证 |
| 前端路由/API 路径变化 | 中 | 全局搜索替换，确保无遗漏 |
| 用户不习惯两个周期概念 | 中 | 前端文案明确区分「目标周期」和「评估周期」 |

## 十二、后续可演进方向

1. **支持多目标周期并行**：比如年度目标 + 季度目标同时存在。
2. **目标与任务/项目打通**：目标关联具体项目或任务进展。
3. **OKR 模式支持**：目标下挂关键结果（KR），支持进度更新。
4. **完全解耦（方案 C）**：目标管理成为独立模块，评估周期可选引用多个目标周期。
