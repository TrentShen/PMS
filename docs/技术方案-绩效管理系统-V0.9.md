# 绩效管理系统 — 技术实施方案

## 一、Context（为什么做、做什么）

### 业务背景
公司（≤100人）当前用 Excel 做绩效管理，目标设定/评分汇总/校准靠人工，**效率低、易错、难追溯**。PRD v1.2 已确认 16 条核心规则（考核周期、3-6-1 分布、串行审批链、业绩1-5分/价值观甲乙丙、互评流程等），参考飞书 People。

### 目标
把「目标设定→自评→互评→上级评估→校准→反馈→公布」全链路线上化，嵌入企业微信工作台，H5 形态交付；HRBP 可发起催办、导出报表；权限严格按"最小可见原则"隔离。

### 本次范围（经用户确认）
**V0.9 精简版 MVP**：先跑通单条主路径——组织架构同步 + 绩效项目创建 + 自评 + 上级评估 + 结果发布。**互评、部门校准+公司级审批、Excel 导入** 放到 Sprint 2。预计 2 人 × 2 个月。

## 二、技术选型（已确认）

| 层级 | 选型 | 说明 |
|------|------|------|
| 前端 | React 18 + Vite + TypeScript + Ant Design Mobile/Ant Design 5 | 企微 H5，PC+移动同一套代码；用 `@wecom/jssdk` |
| 后端 | Python 3.12 + FastAPI + SQLModel + Pydantic v2 | 与现有 JST 技术栈一致，团队熟悉 |
| 数据库 | MySQL 8.0（主）+ Redis 7（缓存/分布式锁/任务队列） | 100人规模单实例足够 |
| 任务调度 | APScheduler（进程内）或 Celery + Redis | 用于每日组织架构同步、截止日提醒 |
| 部署 | **公网 ECS + HTTPS**，Nginx 反向代理，Docker Compose | 域名 `pms.company.com` |
| 鉴权 | 企微 OAuth2 静默授权（免登）| 仅通过企微 PC 客户端 + 移动端入口 |
| 消息推送 | 企微「发送应用消息」API 直调（PRD 3.5.2 已确认不走 WorkBuddy） | 重试3次→降级邮件 |
| 可观测性 | 应用日志 + 操作审计表 + 简单 Grafana/Prometheus（可选） | — |

## 四、数据模型（V0.9 核心表）

只列 V0.9 必需表，其余表（互评、校准、审批、匿名评价）在 Sprint 2 扩展。

```sql
-- 1. 用户与组织（从企微同步）
user (id, wecom_userid, name, avatar, department_ids, position, level,
      leader_userid, hired_at, status, synced_at)

department (id, wecom_dept_id, name, parent_id, leader_userid, order_num)

-- 2. 绩效周期
performance_cycle (id, name, start_date, end_date, stage_json, 
                   result_visible_at, status, created_by, created_at)
                   -- stage_json 存各环节时间线

-- 3. 参与人（项目维度的冻结快照，防止人员变动影响历史）
cycle_participant (id, cycle_id, user_id, leader_userid_snapshot, 
                   dept_path_snapshot, status)
                   -- status: pending/self_done/leader_done/published

-- 4. 目标（V0.9 简化：支持 Excel 导入占位，或手动录入）
objective (id, cycle_id, user_id, title, description, 
           measure_criteria, weight, order_num)

-- 5. 评估记录（一条记录覆盖自评/上级评估两种类型）
evaluation (id, cycle_id, user_id, evaluator_userid, eval_type,
            perf_score, perf_level, value_grade, value_example,
            key_results, comment, submitted_at, status)
            -- eval_type: self/superior
            -- perf_score 0.25 分段校验；perf_level 自动派生

-- 6. 消息日志（降级与重试）
notification_log (id, user_id, channel, title, content, payload_json,
                  status, retry_count, sent_at, error_msg)

-- 7. 操作审计（所有写操作留痕）
audit_log (id, operator_id, action, resource_type, resource_id,
           before_json, after_json, reason, ip, created_at)

-- 8. 数据导出日志（HRBP 导出可追溯）
export_log (id, operator_id, export_type, filter_json, row_count,
            file_name, created_at)
```

**权限实现**：写一个 `scope_filter(user, query)` 依赖函数，所有查询接口强制走它——根据当前用户角色自动在 SQLModel query 上追加 `WHERE` 条件（自己 / 下属 / 本部门 / 全局）。不要在业务代码各处散写权限判断。

## 五、企微集成三件套（V0.9 前置必做）

### 5.1 自建应用申请（阻塞项）
由用户找公司企微管理员操作：
1. 企微管理后台 → 应用管理 → 创建自建应用「绩效管理」
2. 拿到 `corpid`、`agentid`、`secret`
3. 配置「可信域名」= `pms.company.com`，完成 JS 域名校验（放一个 `WW_verify_xxx.txt` 到静态资源根目录）
4. 配置「网页授权及 JS-SDK」可信域名
5. 配置「企业可信 IP」= 应用服务器出口 IP
6. 应用可见范围：全员

### 5.2 OAuth2 免登流程
```
用户在企微点击应用 → 企微带 code 跳到 pms.company.com/callback
  → 后端 /api/auth/callback 用 code 调 gettoken + getuserinfo 换 userid
  → 用 userid 匹配本地 user 表 → 签发 JWT 给前端 → 种 cookie
```

### 5.3 通讯录同步
- 使用「通讯录同步 secret」（独立于应用 secret）调 `department/list` + `user/list_id`
- 每日 02:00 全量同步 + 管理员手动触发按钮
- 关键字段：userid/name/department/direct_leader/position/status

### 5.4 应用消息推送
- 通用封装 `send_wecom_message(user_ids, title, content, link_url)`
- 所有发送前写 `notification_log`，发送后更新状态
- 失败自动重试 3 次（指数退避），连续失败降级邮件

## 七、关键技术点与避坑

1. **企微 `access_token` 2小时过期**：用 Redis 缓存 + 提前 5 分钟刷新，所有调用统一走封装函数。
2. **业绩评分 0.25 分段校验**：前端+后端双重校验，`Decimal(score * 4) % 1 == 0`。
3. **3-6-1 分布校验**：V0.9 不做（没有校准环节），Sprint 2 校准页面实现实时占比显示。
4. **权限最小可见**：所有 list 接口必须过 `scope_filter`，禁止直接 `SELECT *`。代码评审重点。
5. **操作留痕**：校准、结果修改等敏感写操作用 AOP 装饰器自动写 `audit_log`。
6. **冻结快照**：`cycle_participant` 保存 `leader_userid_snapshot` 和 `dept_path_snapshot`，防止考核期间组织调整影响归属。
7. **消息幂等**：同一环节同一用户同一类型提醒每天最多 1 次（Redis setnx key = `notify:{user_id}:{cycle_id}:{stage}:{date}`）。
8. **导出脱敏**：V0.9 无水印，但需要 `export_log` + 只允许 HRBP 角色调用 + 单次导出行数上限（例如 200 行）。
9. **企微应用消息规范**：textcard 模板，标题<128字节、描述<512字节，超出截断。
10. **时区**：服务器和数据库统一 `Asia/Shanghai`，避免截止日期判断偏差。

## 八、验收 / 测试方案

### 端到端测试（每个 Sprint 结束验收）
| 场景 | 预期 |
|------|------|
| 员工在企微工作台点击绩效应用 | 免登，直接看到自己的待办 |
| HRBP 创建一个测试周期，选 5 个参与人 | 5 人收到企微消息通知 |
| 员工提交自评（业绩 4.00 + 价值观 甲 + 事例） | 数据落库，上级收到提醒 |
| 上级完成评估后提交 | 员工状态变为 `leader_done` |
| HRBP 点击"发布结果" | 5 人收到结果公布消息，可查看最终等级 |
| HRBP 导出 Excel | 文件下载成功，`export_log` 有记录 |
| 非本人尝试访问他人绩效 | 403 + audit_log 记录 |

### 自动化
- 单元测试：评分校验、权限 scope_filter、SMART 校验逻辑
- 集成测试：OAuth 流程（用 mock 企微 API）、消息发送幂等
- 性能：100 并发无压力（JMeter 快速验证）

## 九、风险清单

| 风险 | 影响 | 缓解 |
|------|------|------|
| 企微自建应用申请周期拖长 | 阻塞 Sprint 1 | 用户立即推动 IT/管理员，Sprint 0 就启动 |

| 离职人员历史数据保留 | 合规 | `user.status = inactive` 保留不删，查询时按快照查 |
