# PMS · 飞书绩效视角使用改进评估

> 评估时间：2026-06-22  
> 评估范围：`pms/` 前后端完整代码库  
> 参考基准：飞书 People 绩效模块交互体验、PRD v1.2、技术方案 V0.9  
> 验证结果：后端 45 个测试全绿、前端 `tsc --noEmit` 通过、`npm run build` 通过

---

## 一、总体结论：为什么你觉得"不太可用"

代码层面的主流程（目标→自评→互评→上级评估→校准→审批→反馈→发布）在测试环境已经能跑通，但**从真实使用者视角看，系统还差 several "最后一公里"**：

1. **部署配置与真实生产环境脱节**：仓库里的 Nginx/Compose 只配了 HTTP 80 端口，但生产地址是 `https://...:30088/`。若直接按仓库部署，企微 OAuth、可信域名校验、JS-SDK 都会失败，用户连登录都进不来。
2. **流程开关"形同虚设"**：HR 创建周期时可以关闭互评/校准/反馈，但前端根本不读这些开关，后端接口也不拦截。HR 配置了一个"简版考核"，员工看到的仍是完整流程，点进去可能报错或状态混乱。
3. **关键节点没有通知**：周期启动、互评任务分配、审批待办、结果发布等飞书里会自动推消息的场景，当前系统没有触发企微消息。员工不知道要做什么，HR 只能人肉催办。
4. **匿名评价可见性违背 PRD**：代码里把直属上级也纳入了匿名评价可见范围，但 PRD 明确"仅 HR/部门 Leader 可见，直属上级不可见"——这会在真实使用中引发信任危机。
5. **发布前未校验反馈完成**：PRD 要求"结果在反馈完成后对员工可见"，但当前发布只检查有没有最终分数，不检查面谈是否已确认。HR 可能误发布未完成沟通的员工结果。
6. **复杂页面移动端体验仍偏 PC**：校准矩阵、HR 管理台、Leader 评估详情在小屏上字段密集、操作繁琐。飞书绩效的核心优势之一是"手机上也能顺畅填写"，当前差距明显。

**一句话判断**：骨架完整、主路径可跑，但**上线前的部署适配、权限细节、通知触达、移动端体验**这四块如果不修，真实员工用起来会处处碰壁。

## 二、按使用者角色的具体问题

### 2.1 员工："我点进来不知道要干嘛"

| 飞书体验 | 当前 PMS 现状 | 差距 |
|----------|---------------|------|
| 首页清晰展示"当前环节 + 截止倒计时 + 一键进入" | 首页有周期列表，但没有基于当前环节的强引导 | 中 |
| 环节开始/截止自动收到应用消息 | 基本收不到关键通知（除非 HR 手动催办） | **高** |
| 自评填写有进度条、目标与关键成果自动关联 | 目标录入和自评在同一页纵向堆砌，无步骤感 | 中 |
| 移动端表格自动卡片化、输入框适配 | 复杂表格仍要横屏/滚动，小屏输入困难 | **高** |
| 结果公布后收到消息，点击即看最终等级 | 发布节点不触发企微消息，员工需主动进系统刷新 | **高** |

**典型阻塞场景**：员工在手机上打开自评页，录入 3-5 条目标后下滑填自评，屏幕小、内容长，容易误触；提交后也不知道下一步该做什么。

### 2.2 Leader（直属上级/部门负责人）："评估一个人要滑很久"

| 飞书体验 | 当前 PMS 现状 | 差距 |
|----------|---------------|------|
| 一页式摘要：目标/自评/互评/历史 分区折叠/锚点 | `LeaderEvalDetail` 单页纵向堆砌所有内容 | 中 |
| 互评汇总展示打分分布 + 手松手紧提示 | 已实现三维度汇总和手松手紧统计 | ✅ 基本对齐 |
| 匿名评价严格隔离，避免直属上级看到 | **代码错误地把直属上级纳入可见范围** | **阻塞** |
| 校准会支持在线协同、实时分布预警 | 已有矩阵热力图和 3-6-1 预警 | ✅ 基本对齐 |
| 移动端查看下属列表和提交评估 | 表格列多、弹窗表单项多，小屏操作繁琐 | **高** |

**典型阻塞场景**：Leader 在手机上给下属写评估，页面很长没有锚点；校准矩阵列数多，小屏需要横屏才能看清；如果员工写了匿名评价，Leader 可能意外看到，违反制度。

### 2.3 HRBP/绩效管理员："配置完流程，但系统不听我的"

| 飞书体验 | 当前 PMS 现状 | 差距 |
|----------|---------------|------|
| 创建考核时可开关各环节，系统按开关控制界面和流程 | `enable_*` 字段存在，但**前端不读、后端不拦截** | **阻塞** |
| 一键启动后自动通知全员 | 启动节点未触发企微消息 | **高** |
| 各环节进度看板 + 自动催办 | 催办需手动，截止提醒只按通用逻辑扫描 | 中 |
| 结果发布前校验所有沟通已完成 | 发布只校验 final_perf_score，不校验反馈确认 | **阻塞** |
| 导出 Excel 包含完整三维度价值观 | 已输出三维度列 | ✅ 基本对齐 |

**典型阻塞场景**：HR 为了月底启动会，先关闭互评和校准，只跑"目标+自评+上级评估+发布"。但员工端仍然显示"邀请互评人"按钮，点进去后端接口也没有拒绝，导致数据状态和流程混乱。

### 2.4 CEO/HR 审批人："审批待办不会主动推给我"

| 飞书体验 | 当前 PMS 现状 | 差距 |
|----------|---------------|------|
| 审批待办自动推送应用消息 | 当前校准提交后不会自动通知 CEO/HR | **高** |
| 驳回时自动通知被驳回人并附原因 | 驳回记录落库，但**不触发消息** | **高** |
| 手机端可一键批准/驳回 | 校准页面移动端复杂，审批操作不便 | 中 |

## 三、阻塞级问题清单（不修复不建议上线）

| 编号 | 问题 | 现象 | 影响 | 涉及文件 | 修复方向 |

| B-2 | 企微 `agentid` 为空时消息发送崩溃 | `wecom.py` 中 `int(settings.wecom_agentid)` 在空字符串时抛 `ValueError` | 配置遗漏时所有通知/催办接口 500 | `backend/src/pms/services/wecom.py:267,292,310` | 发送前校验 `wecom_agentid/corpid/secret`；缺失时记录 error 并优雅返回 |
| B-3 | 匿名评价可见性违背 PRD | `peer.py:524` 把 `direct_leader` 纳入 `can_see_anon` | 直属上级能看到匿名评价，引发信任危机 | `backend/src/pms/api/v1/peer.py:523-524` | 移除 `direct_leader`，仅保留 `hrbp/super_admin/dept_leader` |
| B-4 | 流程开关 `enable_*` 未生效 | 创建/更新周期时可配置开关，但前端不读、后端业务接口不拦截 | HR 配置的流程与实际用户看到的流程不一致 | `backend/src/pms/api/v1/evaluations.py`、`peer.py`、`calibration.py`、`feedback.py`；前端各页面 | 后端写接口入口检查 `cycle.enable_*`；前端根据开关隐藏入口/按钮 |
| B-5 | 发布前未校验反馈确认 | `cycles.py:publish_cycle` 只检查 `final_perf_score` | 可能发布未完成面谈沟通的员工结果，与 PRD"反馈后可见"冲突 | `backend/src/pms/api/v1/cycles.py:214-223`、`feedback.py` | 当 `enable_feedback=True` 时，校验所有参与人 `FeedbackRecord.confirm_status != pending` |

## 四、高优先级问题清单（严重影响可用性）

## 五、与飞书绩效的核心差距（功能层面）

以下属于"体验差距"而非阻塞项，但若想在月底启动会上让用户觉得"好用"，建议至少做前三项：

| 能力 | 飞书 People | 当前 PMS | 建议 |
|------|-------------|----------|------|
| **富文本自评** | 支持 | 普通文本框 | 长期：接入富文本编辑器 |
| **目标与关键成果自动关联** | 填写自评时可引用目标 | 目标与自评分离 | 中期：自评页增加"关联目标"选择 |
| **面谈预约** | 可预约时间、自动提醒 | 只能线下沟通后补记录 | 长期：增加预约能力 |
| **智能互评分配** | 系统根据协作关系推荐互评人 | 员工自选 + Leader 审核 | 长期：基于部门/项目关系推荐 |
| **绩效趋势/人才盘点** | 历史趋势图、九宫格 | 仅基础历史列表 | PRD V1.2 规划 |
| **结果与薪酬/晋升联动** | 可配置 | 未做 | PRD 明确首次不做 |

## 六、建议立即执行的 10 件事（未来 1-2 周）

按优先级排序：

1. **补齐生产 HTTPS 与端口配置**：让仓库配置与真实生产地址一致，避免企微登录失败。
2. **修复匿名评价可见性**：移除 `direct_leader`，严格符合 PRD。
3. **让流程开关真正生效**：前后端联动，HR 关闭的环节用户看不到、调不通。
4. **修复发布前的反馈确认校验**：确保"先沟通、后公开"。
5. **补齐关键节点企微通知**：启动、互评任务、审批待办、结果发布必须推消息。
6. **增加企微配置缺失的防御性处理**：避免配置遗漏时接口 500。
7. **修复 `feedback.py` N+1 查询**：参与人多时不会拖慢页面。
8. **匿名评价三维度化**：与主评估流程维度一致。
9. **实现邮件降级通道**：企微消息失败有兜底。
10. **优化移动端关键页面体验**：校准弹窗步骤化、HR 管理台手机端卡片化、Leader 评估详情增加锚点/折叠。

## 九、给负责人的 3 个关键决策

1. **上线范围是否包含互评/校准？**
   - 若时间紧，建议先关闭 `enable_peer_eval` 和 `enable_calibration`，只跑"目标+自评+上级评估+反馈+发布"的最小闭环。但前提是先把 B-4（流程开关生效）做了。

2. **CEO 审批是否允许 HR 代行？**
   - 当前代码已支持 HR 代 CEO 审批，适合 100 人公司。若要坚持 CEO 本人审批，需提前准备 CEO 企微测试账号并配置角色。

3. **匿名评价是否必须三维度化？**
   - 建议先修复可见性（B-3），维度问题（H-3）可放到上线后第一周再做，因为匿名评价是"补充信息"，不影响主流程。
