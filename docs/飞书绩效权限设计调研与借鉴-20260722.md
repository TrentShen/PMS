## 一、飞书绩效的权限模型要点

### 1.1 角色与入口
- 每个考核项目只有 4 种参与角色：**员工、Leader、校准人、HRBP**，外加项目级的**绩效管理员**（配置方案、环节、分布规则）。
- **校准权限自动按管理范围配置给管理者**，校准任务按项目分组、按校准范围分发；没有校准权限的人**看不到校准入口**（来源：[飞书绩效校准人手册](https://www.feishu.cn/content/SE1jdFoMworroxxpMCRcI5mbnCT)）。
- 各环节流转由系统驱动：自动提醒当前环节的角色参与，环节完成自动流转下一环节（[快速了解飞书绩效](https://www.feishu.cn/hc/zh-CN/articles/296219674493-%E5%BF%AB%E9%80%9F%E4%BA%86%E8%A7%A3%E9%A3%9E%E4%B9%A6%E7%BB%A9%E6%95%88)）。

### 1.2 各环节的数据可见性（核心）

| 角色/环节 | 能看到什么 |
|---|---|
| **360 环评人（互评人）** | 被评估人基本信息 + 评估表；评估**依据 OKR 和重点工作**进行（即环评人可看到被评估人的目标/重点工作内容，作为打分依据）。看不到他人的评价。 |
| **被评估人（员工）** | 自己的 OKR、自评；360 评价内容**对本人匿名且不公开明细**；最终结果需经"结果确认"环节才可见，可提异议。 |
| **上级（Leader）** | 下属 OKR、重点工作、自评、360° 环评汇总 + 系统对每位环评人"**手松手紧**"的智能提示。 |
| **校准人** | 上下文最全：360° 评估结果、工作总结、自评、OKR、**历史绩效详情**，左侧栏参考区完整呈现；可看同阶段其他校准人的校准状态。 |
| **HRBP/管理员** | 全环节进度、分布报表、催办；按管理范围过滤。 |

### 1.3 值得注意的机制设计
1. **多校准人协同**：同阶段多位校准人同时校准时，可互相看到校准状态；已校准结果被他人修改时**弹窗提醒 + bot 通知**。
2. **分布规则硬校验**：管理员可配置强制/建议分布规则；校准人点"完成校准"时系统校验，不符合则弹窗提示调整或确认提交；异常指标在图表中高亮、可点击下钻到人员列表。
3. **校准原因必填**（管理员可配置开关）：修改上级打出的结果必须填原因，原因进变更记录。
4. **逾期策略可配置**："任务逾期后允许首次提交"开关；校准截止后**结果锁定不可改**，只能补点"完成校准"。
5. **批量导入校准结果**：下载模板 → 填写 → 上传 → 核对调整详情 → 确认导入。
6. **差异分析报表**：对比不同评估角色（自评 vs 360 vs 上级）、不同评估项、不同周期的差异，极端差异人员标红、可下钻。
7. **字节跳动实践**：流程为 OKR → 360 邀请 → 360 评估 → 360 确认 → 上级评估 → 校准 → 绩效确认；校准会同时也是"人才盘点"场景（[对标考察网](http://www.duibiao.org/2022/news_0222/2719.html)）。

## 二、PMS 现状（代码级梳理，2026-07-22）

### 2.1 已对齐飞书的点
- **互评对上级匿名聚合**：peer/summary 只返回平均分、价值观分布、评语列表，不含评价人姓名；"手松手紧"分析里评价人编号为"评价人1/2/3"（`peer.py:467-582`）。✅
- **被评人看不到互评内容**：本人访问 peer/summary 被明确拒绝（`peer.py:475-480`）。✅
- **互评名单员工提名 + 上级审批**：与字节"360 邀请 → 360 确认"一致（`peer.py:105-183, 286-317`）。✅
- **手松手紧提示**：已实现。✅
- **校准原因必填 + 变更记录**：`CalibrationRecord` 记新旧值、理由、操作人。✅
- **校准分布预警**：3-6-1 热力图预警。✅（但只是预警，非硬校验）
- **结果先沟通后公开**：反馈确认前 final 分数对员工隐藏（`evaluations.py:138-152`），理念等同飞书"结果确认"环节。✅
- **HR 部门成员绩效数据对其他角色不可见**（`scope.py` 剔除 HR 部门），保护 HR 自身数据。✅
- **校准按 scope 自动分权**：dept_leader 见本部门树、hrbp 按 scope、超管全局，约等于飞书"校准权限自动配置给管理者"。✅

### 2.2 主要差距（借鉴价值从高到低）

| # | 差距 | 飞书做法 | PMS 现状 | 借鉴建议 |
|---|------|----------|----------|----------|
| 1 | **互评人看不到被评人的目标/工作内容** | 环评人"依据 OKR 和重点工作"评估 | 互评人只看到姓名+职位+周期名，纯主观打分（`peer.py:360-387`、`PeerTasks.tsx`） | 互评表单页增加被评人的**目标 + 自评（可选）**只读区，让评价有据可依；可做周期级开关 |
| 2 | **校准人上下文太少** | 校准人左侧栏：360 结果、自评、工作总结、OKR、历史绩效 | 校准视图只有上级初评分/等级/价值观（`calibration.py:60-76`） | 校准矩阵行点击展开详情抽屉：目标/自评摘要/互评聚合/历史绩效（复用 detail 接口数据） |
| 3 | **多校准人协同缺失** | 同阶段校准人状态可见；结果被改弹窗 + bot 通知 | 单人单范围假设，无协同感知 | 100 人团队优先级低；可先加"该行最后校准人+时间"列，避免覆盖 |
| 4 | **分布规则只预警不校验** | 完成校准时硬校验强制/建议规则 | 3-6-1 仅热力图预警，提交不拦截 | 提交校准时增加分布校验：不符则弹窗列差异，需确认理由（软校验即可） |
| 5 | **逾期策略不可配** | "逾期允许首次提交"开关；截止后锁定 | 无截止时间概念 | 中期：周期加各环节截止时间 + 逾期锁定 |
| 6 | **批量导入校准结果** | 模板下载→上传→核对→导入 | 只能逐条改 | 低优先：100 人团队手动够用；历史绩效导入已类似 |
| 7 | **差异分析报表** | 自评 vs 360 vs 上级评分差异矩阵，极端标红下钻 | 只有单点"手松手紧" | HR 看板加"自评 vs 上级"散点/差异列表，识别自评虚高/上级压分 |

### 2.3 调研中顺带发现的现有问题（与飞书无关，建议另行处理）
1. `evaluations.py:365-372`：注释写"HR 只能看不能代评"，但 `can_act_as_superior` 实际允许 hrbp/super_admin 提交上级评估——实现与注释不一致，需确认意图。
2. `trend.py:93-106`：部门趋势接口未按 `hrbp_scope_dept_ids` 过滤，任何 hrbp 可见全公司部门均分——scope 漏洞。
3. 互评"匿名"是展示层匿名、DB 实名，审计 `submit_peer_evaluation` 的 resource_id 含 task_id，理论上可经 DB 反查评价人——合规上需知晓并控制 DB 访问。
4. 2026-06-22 评估的 B-4（`enable_*` 流程开关未生效）需确认当前状态，本次未重新验证。

## 三、建议行动（按 ROI 排序）

1. **互评表单增加"被评人目标"只读参考区**（差距 1）——改动小（my-tasks 接口加返回目标字段 + PeerTasks 页展示），对评价质量提升最直接。是否同时展示自评建议做成周期配置项，默认只展示目标。
2. **校准页增加人员详情抽屉**（差距 2）——校准人不用跳页就能看到目标/自评/互评聚合/历史绩效。
3. **校准提交加分布软校验**（差距 4）——弹窗提示 + 理由确认，复用现有 3-6-1 计算。
4. **修复 2.3 的两处权限问题**（HR 代评注释不一致、部门趋势 scope 漏洞）——纯后端小改动。
5. 差距 3/5/6/7 列入 V1.0  backlog。
