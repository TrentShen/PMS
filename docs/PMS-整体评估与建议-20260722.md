# PMS 整体评估与建议（按公司规模适配）

> 评估时间：2026-07-22
> 公司规模前提：100 人内小团队（AGENTS.md 项目身份卡），单层/浅层汇报结构，HR 人力有限（预计 1-2 人兼管绩效）
> 评估依据：代码库现状、飞书绩效权限调研（2026-07-22）、6-22 使用改进评估、7-22 部署验证结果

---

## 一、总体结论

**系统的"骨架"对 100 人团队来说已经超标完成，当前瓶颈不在功能缺失，而在"运转顺畅度"**：权限有两处小漏洞、通知触达和企微入口还没闭环、部署仍依赖人工密码。对 100 人公司，正确策略是**冻结新功能、打磨现有闭环、把运营动作（催办/提醒/入口）做扎实**，而不是继续对标飞书加能力。

---

## 二、分维度评估

### 2.1 功能完整度：★★★★☆（对规模而言偏"重"）
- 主流程全覆盖：目标 → 自评 → 互评 → 上级评估 → 校准 → HR/CEO 审批 → 面谈反馈 → 发布，另有试用期管理、匿名评价、历史绩效导入、绩效趋势。
- 对 100 人团队**偏重的部分**（建议降级使用而非再开发）：
  - 校准的 HR → CEO 两级审批链：100 人公司通常 HR 负责人终审即可，CEO 代批已支持，建议直接简化为一审。
  - 匿名主动评价（3D 化）：使用频率存疑，保留但不投入。
  - 多校准人协同、九宫格人才盘点、薪酬联动：**明确不做**（见飞书调研差距 3/5/6）。

### 2.2 权限与数据可见性：★★★★☆（有两处必修漏洞）
- 互评匿名聚合、被评人不可见互评、名单上级审批、校准 scope 分权、HR 部门数据隔离——这些核心设计与飞书一致，**不用改**。
- 必修（工作量均 < 半天）：
  1. `trend.py` 部门趋势未按 `hrbp_scope_dept_ids` 过滤，任何 HRBP 可见全公司部门均分。
  2. `evaluations.py` "HR 只能看不能代评"注释与实现矛盾，需确认本意后统一。
- 值得借鉴的一处增强：互评表单加"被评人目标"只读参考区（飞书环评人按 OKR/重点工作打分），对评价质量提升最直接。

### 2.3 技术健康度：★★★☆☆
- 好的：前后端类型严格、CI 三道检查、契约检查脚本、审计日志完整、迁移可一键建库。
- 欠账：
  - eslint 未安装（`npm run lint` 形同虚设）；
  - 前端 bundle ~3MB 未拆分（企微 H5 首屏偏慢，100 人可接受但值得一次 manualChunks）；
  - expect 部署脚本会误报错误（7-22 已发生一次），SSH 密钥未被服务器授权，部署依赖人工密码；
  - S-6（人事助手接口权限）仍待验证；
  - 6-22 提出的 B-4（`enable_*` 流程开关未生效）本次未复验，需确认现状。

### 2.4 体验（企微 H5）：★★★★☆（7-22 UI 改造后）
- 7-22 已完成设计令牌 + Linear 布局 + 六核心页面移动端卡片化/底部操作栏，最痛的移动端短板已补齐。
- 待真实用户走查验证（建议 HR + 2-3 名 Leader 在手机上完整跑一遍自评→评估→校准）。

### 2.5 运营闭环：★★☆☆☆（当前最大短板）
100 人公司绩效系统成败 90% 在运营，不在功能：
- 企微应用主页 URL 仍是 `http://10.222.4.38`（7-22 已做 301 兜底，但入口本身要改）。
- 关键节点通知需确认是否全部生效（启动/互评任务/审批待办/发布）。
- 无 UAT 验证账号矩阵（HRBP/Leader/员工各一）。

## 三、建议清单（按优先级）

### 立即做（本周，都是小改动）
1. 修 `trend.py` 部门趋势 scope 漏洞。
2. 确认并统一"HR 能否代评"的逻辑与注释。
3. 企微后台应用主页 URL 改为 `https://shanghai.idc.matrixorigin.cn:30088/`。
4. 复验 `enable_*` 流程开关现状（关互评/校准的简版考核能否真正跑通）——100 人团队很可能只想跑简版。
5. 装 eslint 进 devDependencies，让 `npm run lint` 可用。

### 近期做（2-4 周）
6. 互评表单加"被评人目标"只读参考区（默认展示目标，自评是否展示做成周期开关）。
7. 校准页人员详情抽屉（目标/自评/互评聚合/历史绩效，复用 detail 接口）。
8. 校准提交加分布软校验（不符弹窗 + 理由确认，复用 3-6-1 计算）。
9. 部署自动化收尾：请运维把 `~/.ssh/id_deploy.pub` 加入服务器 `authorized_keys`；排查 expect 脚本误报。
10. 通知全链路验证：用测试周期把启动→互评→审批→发布的消息都触发一遍。
11. HR + Leader 手机端 UAT 走查。

### 明确不做（与规模不匹配）
- 多校准人实时协同、校准逾期策略配置、批量导入校准结果
- 九宫格人才盘点、薪酬/晋升联动、富文本自评、面谈预约
- 智能互评推荐（100 人互相都认识，员工自选 + 上级审批足够）

### 流程建议（非系统）
- 校准审批简化为 HR 一审，CEO 只在异议时介入。
- 首次正式跑周期建议用"目标+自评+上级评估+反馈+发布"简版（关互评和校准），降低首次推广阻力——前提是第 4 项验证通过。
